### What was built

The second stage of the screening pipeline â€” a real-time, adaptive AI interview with voice output and speech-to-text input.

**Interview chat UI**
- Full-screen chat interface (`/jobs/:id/apply/:candidate_id/interview`) rendered via `interview.html.erb`
- Fixed bottom input area with auto-growing textarea; submit on Enter (Shift+Enter for new line)
- Message thread rendered with `_ai_message.html.erb` and `_candidate_message.html.erb` partials
- Turbo Streams append new messages in real time without a full page reload

**AI interview conductor (`InterviewConductor` service)**
- Checklist-driven adaptive loop: questions generated by Claude based on job scenario and CV evaluation gaps
- Tracks answered checklist items per conversation; stops when all items are resolved or max turns reached
- On completion, calls `SummarizeInterview` service and saves a structured JSON summary (score, overall pass/fail, written summary) to `candidate.interview_summary`
- Routing logic in `ApplicationsController#create_message`: creates candidate and AI messages, checks for completion, returns Turbo Stream response

**Turbo Stream completion flow**
- On final answer: `interview-input` replaced with an empty div (hides the form), `interview-frame` replaced with `_interview_complete.html.erb`
- Inline summary card shows score (`/100`), pass/fail verdict with icon, and written summary text
- "View Full Result" button links to `/jobs/:id/result?candidate_id=:id`

**Per-message TTS (ElevenLabs)**
- Each AI message has a speak button (avatar circle) with a CSS-driven state machine: `idle â†’ loading â†’ playing â†’ paused â†’ ended`
- State stored as `data-speech-state` attribute; all icon visibility controlled by CSS attribute selectors â€” no JS class toggling
- Audio and word timing data cached in a `WeakMap` keyed on the button element for replay without re-fetching
- ElevenLabs character-level alignment parsed into word objects; 40 ms interval tracks `audio.currentTime` for word-by-word highlighting (`.word-current` / `.word-spoken` CSS classes)
- Auto-plays the last AI message on page load; auto-plays each new AI message appended by Turbo Stream
- Play / pause / replay supported; replay reuses cached audio (seeks to 0, re-renders word spans)

**Speech-to-text dictation (Web Speech API)**
- Mic button in the input area toggles `SpeechRecognition` with `continuous: true` and `interimResults: true`
- Interim results shown live in textarea; only final transcripts committed to `committed` string on `onend`
- Mic state reflected via button border color and animated red dot; stops automatically on form submit
- Graceful fallback: alerts user if browser does not support the Web Speech API

**Stimulus `interview` controller**
- `MutationObserver` on `#messages` (`subtree: false`) detects newly appended AI messages: resets loading spinner, clears textarea, focuses input, starts TTS
- `#setLoading` shows/hides spinner and submit button
- `#scrollToBottom` smooth-scrolls to latest message after every mutation

**ElevenLabs TTS prototype (`/dev/text_to_speech`)**
- Standalone dev-only sandbox for testing TTS ahead of the interview feature
- Rails controller proxies requests to ElevenLabs keeping the API key server-side
- Word highlighting and audio playback via Web Audio API

## [Unreleased] â€” Stage 1: CV Upload & AI Screening (Proof of Concept)

### What was built

The first working end-to-end stage of the AI screening pipeline:

**CV Upload flow**
- Candidate submits their name, email, and a CV file (PDF, Markdown, or plain text) via the application form
- Active Storage handles file upload and attachment to the `Candidate` record
- On submission the app immediately redirects to a live status page that polls for progress

**Background processing pipeline**
- `ProcessCvJob` downloads the attached file and extracts its text content:
  - PDFs are sent to the Claude API (claude-opus-4-6) which returns clean Markdown
  - Plain text / Markdown files are read directly
- `EvaluateCvJob` sends the extracted text plus the job's evaluation scenario to Claude, which returns a structured JSON result (overall pass/partial/fail, score 0â€“100, summary, skill gaps, follow-up questions)
- Both jobs include retry logic for transient Anthropic API errors (rate limits, server errors, connection failures) and discard on non-retryable billing/auth errors

**State machine (`Candidate` status)**
- Powered by Statesman via the `statesman_scaffold` gem
- States: `cv_processing â†’ ready_for_evaluating â†’ evaluating â†’ evaluated` (plus future `interviewing`, `completed`, `accepted`, `rejected`)
- The `status` column is kept in sync after every transition via an `after_transition` callback
- `inquiry_attrs` adds nil-safe predicate methods: `candidate.status.cv_processing?`, `candidate.status.evaluated?`, etc.

**Job model**
- Added `Job.active` / `Job.closed` named scopes
- `job.status.active?` predicate available via `inquiry_attrs`
